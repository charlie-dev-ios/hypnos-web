# Research: 食材情報閲覧

**Date**: 2026-02-03
**Feature**: 006-ingredient-list

## 既存コードベースの調査結果

### 1. 食材データの現状

**Decision**: 新規で食材マスターデータを作成する

**Rationale**:
- 現在の `recipes.json` には食材名と使用数量のみが含まれている
- 食材のエナジー値を持つマスターデータは存在しない
- 各食材の基本エナジー値を含む新しいデータファイルが必要

**Alternatives considered**:
- レシピデータから食材を動的に抽出する → エナジー値が取得できないため却下

### 2. 食材ページの現状

**Decision**: 既存のプレースホルダーページを実装で置き換える

**Rationale**:
- `/ingredients` ページは既に存在し、「準備中です」と表示している
- ナビゲーションリンクも設定済み（navigation-links.ts）
- 既存のルーティング構造を活用できる

### 3. コンポーネント設計パターン

**Decision**: recipes-page-contentと同様のServer/Client分離パターンを採用

**Rationale**:
- プロジェクトの既存パターンに従う
- Server Componentでデータ取得、Client Componentで表示を担当
- テスタビリティが高い

### 4. データ構造

**Decision**: 以下のスキーマで食材データを定義

```typescript
// Ingredient Schema
{
  id: number,           // 一意識別子
  name: string,         // 食材名（例: "つやつやアボカド"）
  energy: number,       // 基本エナジー値
  imageUrl: string,     // アイコン画像パス（optional）
}
```

**Rationale**:
- issue30の受け入れ基準に対応（アイコン、名前、エナジー値）
- 既存のレシピスキーマと整合性がある

### 5. 食材データソース

**Decision**: 静的JSONファイルとして `src/content/ingredients/ingredients.json` に配置

**Rationale**:
- recipes.jsonと同様のパターン
- 現時点でAPIは不要（静的サイト）
- 将来的にAPI化する場合も移行しやすい
